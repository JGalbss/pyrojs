# How pyrojs works

A fireworks engine has two jobs that pull in opposite directions: it has to look
like *real fireworks* — shells that rise, arc, and break into trailing stars that
fall under gravity — and it has to do that for tens of thousands of particles at
60fps. This is a tour of how pyrojs does both, and where Effect-TS fits without
costing frames.

## The particle store is Structure-of-Arrays

The naive design is an array of particle objects. That is a disaster for a hot
loop: every object is a separate heap allocation, the garbage collector chases
them forever, and iterating them thrashes the cache.

pyrojs instead uses **Structure-of-Arrays (SoA)**. Each attribute lives in its own
typed array, and a particle is just an index `i`:

```ts
px: Float32Array   // position x
py: Float32Array   // position y
vx, vy: Float32Array  // velocity
life, maxLife: Float32Array
r, g, b: Uint8ClampedArray  // color
flags: Uint8Array  // glow / streak / twinkle / color-shift / ghost
```

Dead particles are removed with **swap-remove**: the last live particle is copied
into the dead slot and the count is decremented. The active range stays densely
packed in `[0, count)` with no holes, no free list, and zero per-frame
allocation. The store grows geometrically up to a hard ceiling, then gracefully
drops new spawns instead of allocating without bound.

## The physics is plain Newton

Each frame advances every particle by `dt` seconds. Velocity gets gravity, wind,
and air drag; position integrates velocity:

```ts
const damp = Math.exp(-drag * dt)        // exponential air resistance
vx = vx * damp + wind * dt
vy = vy * damp + gravity * dt
px += vx * dt
py += vy * dt
life -= dt
```

A few details matter for *correctness*, not just speed:

- **Drag is exponential**, `v · e^(−k·dt)`, not `v · (1 − k·dt)`. The exponential
  form is stable at any frame rate and never flips velocity negative on a big step.
- **Turbulence scales with √dt.** Turbulence is a random-walk impulse, and the
  variance of a random walk grows with time, so the per-step magnitude has to be
  proportional to `√dt` to look the same at 60Hz and 144Hz.
- **dt is clamped.** A backgrounded tab can hand you a one-second delta; clamping
  to ~50ms stops the show from teleporting on the first frame back.

## Rendering: additive glow and time-based trails

Overlapping sparks should *add* light, so the renderer draws with
`globalCompositeOperation = "lighter"`. Each star is a cached radial-gradient
**sprite** drawn with `drawImage` — far cheaper than per-particle `shadowBlur`,
and the sprite cache is keyed by a quantized color so it stays tiny even when an
effect sweeps through a color transition.

Motion **trails** come from fading the previous frame instead of clearing it. The
subtlety is that fading a *fixed fraction per frame* makes trail length depend on
frame rate. So the fade is time-based: if `trail` is the fraction retained over
one 60fps frame, the per-frame retention at an arbitrary `dt` is

```ts
retention = trail ^ (dt / (1 / 60))
```

On a transparent overlay we can't paint a translucent background (it would tint
the page), so the fade uses `destination-out` to subtract alpha instead.

## Effect-TS is the backbone — but not the inner loop

Everything *around* the kernel is Effect. Configuration is validated once with
`Schema`; failures are typed `Data.TaggedError` channels. The engine is a
`Scope`-owned program: the canvas context and the DOM observers are acquired as
scoped resources, and the render loop and the autopilot are `forkScoped` fibers,
so tearing the engine down interrupts everything and runs every finalizer. State
lives in `Ref`s; the choreography DSL is built on `Schedule`.

The one rule: **never `yield*` per particle.** A `requestAnimationFrame`-driven
fiber awaits each frame, and the entire numeric kernel — integrate N particles,
emit the draw calls — runs inside a single `Effect.sync` per frame. That is the
idiomatic way to host a CPU-bound kernel in Effect: you get typed errors,
structured concurrency, and resource safety everywhere it helps, and raw
typed-array speed where it counts. The Effect code scores 100/100 on agent-doctor.

## Multi-stage breaks are sub-shells

A crossette that splits into little crosses, or a shell that breaks into a shell
of shells, can't be faked with one burst. pyrojs models a **secondary spark
shell**: a sub-projectile with its own fuse that flies out, trails, and then
detonates into its own effect. The same mechanism powers the rising shell (a
fuse that triggers at apex) and crossette, multibreak, and roman candle.

## Shapes and image fireworkify

A shape burst is just a clever choice of initial velocity. If you want the stars
to trace a figure, set each star's velocity to its target offset and crank up the
drag so they decelerate *into* position:

```ts
velocity = shapeOffset * power   // + high drag to settle into the shape
```

Hearts and stars come from parametric curves. Arbitrary images and SVGs come from
sampling: rasterize the source small, keep the opaque pixels, and — crucially —
reduce the colors to a handful of dominant **layers** with median-cut
quantization. Without that step, additive blending smears a full-color image into
a gray cloud; with it, the break reads as crisp posterized regions, like the
picture pixelized.

## A note on performance

The kernel is the whole point, so it's worth measuring. On an M-series laptop the
physics step (no drawing) costs:

| live particles | step / frame |
| ---: | ---: |
| 10,000 | ~0.08 ms |
| 50,000 | ~0.8 ms |
| 100,000 | ~0.8 ms |

At 50,000 particles the simulation uses about 5% of a 60fps frame budget — the
rest is free for the GPU to draw. In practice, rendering is the ceiling, not the
math, which is exactly where you want a particle engine to be. The loop also
pauses entirely when the tab is hidden, and the whole show is deterministic given
a `seed` (a mulberry32 PRNG), so recordings and tests reproduce byte-for-byte.
